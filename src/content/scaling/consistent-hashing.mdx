## Introduction
Consistent hashing is a special kind of hashing that allows a system to scale its resources (like cache servers or database nodes) without requiring a massive data reshuffle when nodes are added or removed.

## Why It Matters
In traditional hashing, adding a single server can cause 90% of your data to move to new locations. This causes "Cache Stampedes" or massive network spikes. Consistent hashing reduces this movement to a tiny fraction ($1/n$), keeping your system stable during scaling events.

## Problem It Solves
Traditional `hash(key) % N` routing is fragile. If $N$ changes, almost every key maps to a different server. In a distributed cache, this causes a 100% miss rate; in a database, it causes massive data migration. Consistent hashing provides a stable mapping that survives cluster resizing.

## Core Concept: The Hash Ring
Instead of mapping keys to a linear range of buckets, we map both **keys** and **servers** onto a 360° circular "Hash Ring".
- **Servers** are hashed and placed at specific points on the ring.
- **Keys** are hashed and placed on the ring. 
- A key is assigned to the first server it encounters while moving **clockwise** around the ring.

### Virtual Nodes (VNodes)
To ensure even distribution, we map each physical server to multiple points on the ring using "Virtual Nodes". This prevents "Hot Spots" and allows powerful servers to handle more load by giving them more VNodes.

## Real-World Usage
### Level 1: Foundational (Caching)
**Memcached** and **Redis** clusters use consistent hashing to ensure that adding a new cache server doesn't invalidate the entire cache. If you add a 4th server, only ~25% of existing keys need to be fetched again.

### Level 2: Enterprise (Global Scale)
- **Discord** uses consistent hashing to assign users to specific chat gateway servers. 
- **Cassandra & DynamoDB** use it to determine which partition of data lives on which physical disk.
- **Akamai CDN** uses it to route client requests to the nearest edge server without losing the cache context.

## Interview Notes
<InterviewAnswer>
### Describe Consistent Hashing and why we need Virtual Nodes.

1. **The $1/N$ Guarantee**: Explain that when a cluster grows from $N$ to $N+1$ nodes, only $1/(N+1)$ of the data needs to move. Traditional modulo hashing would move nearly 100%.
2. **The Hash Ring**: Describe placing both keys and servers on a 0 to $2^n-1$ circular space. A key is owned by the first node encountered moving clockwise.
3. **Partitioning**: Each node is responsible for the arc of the ring between it and its counter-clockwise neighbor.
4. **Virtual Nodes**: Physical nodes are represented multiple times on the ring. This solves two problems: non-uniform distribution (hotspots) and heterogeneous hardware (making the ring scale with server capacity).
5. **Node Failure**: When a node fails, its "slice" of the ring is automatically inherited by its clockwise neighbor, ensuring high availability with localized impact.
</InterviewAnswer>

## Common Questions (Q&A)
<FAQAccordion items={[
  {
    question: "What happens if a node fails in the hash ring?",
    answer: "The keys that were mapped to the failed node are automatically 'passed' clockwise to the next available node on the ring. Only the data from the failed node needs to be re-replicated or re-cached."
  },
  {
    question: "How does Consistent Hashing compare to Rendezvous Hashing?",
    answer: "Both solve the remapping problem. Consistent hashing is generally faster for lookups ($O(\\log n)$), while Rendezvous hashing (Highest Random Weight) is $O(n)$ but provides even better distribution and doesn't require a ring structure."
  },
  {
    question: "Why use Virtual Nodes instead of just physical nodes?",
    answer: "Without Virtual Nodes, servers are just single points on the ring. This can lead to uneven data distribution. Virtual Nodes allow one server to appear in multiple locations, ensuring a much smoother and fairer distribution of data."
  },
  {
    question: "Is Consistent Hashing only for caches?",
    answer: "No! While famous for caching (like Redis), it's equally important for distributed databases (Cassandra), load balancers (mapping clients to servers), and even decentralized file systems."
  }
]} />
