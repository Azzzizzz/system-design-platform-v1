# Consistent Hashing

Consistent hashing is a special kind of hashing such that when a hash table is resized, only $K/n$ keys need to be remapped on average, where $K$ is the number of keys, and $n$ is the number of slots.

## The Rehash Problem

In a standard distributed system, you might route requests using a simple modulo: `server = hash(key) % total_servers`.

This works... until you need to scale. 
- If you have **10 servers** and add **1 more**, almost **100% of your requests** will hash to a different server.
- In a caching system, this means a total cache miss storm.
- In a stateful system, it means massive data migration.

## The Hash Ring Solution

Instead of mapping to a range of numbers, we map both **Requests** and **Servers** to a circular 360° ring.

### Key Advantages

1. **Minimal Disruption**: When a node is added, it only "steals" keys from its immediate counter-clockwise neighbor.
2. **Scalability**: Seamlessly add or remove servers with predictable impacts.
3. **Partitioning**: Each server is responsible for a "slice" of the hash space.

---

## Real-World Application

### Level 1: Foundational (Caching)
A single **Memcached** or **Redis** cluster usually employs consistent hashing. By mapping keys to a hash ring, adding a third cache server only requires remapping 33% of the keys, rather than 100%. This prevents a "Cache Stampede" where the database is suddenly overwhelmed because the cache was invalidated.

### Level 2: Enterprise (Global Content Delivery)
**Akamai** and **Cloudflare** use consistent hashing at a global scale. When an edge node in London goes down, only the traffic that was specifically routed to that node is reassigned to the nearest healthy nodes. The rest of the global traffic remains unaffected, ensuring massive stability.

## Implementation Patterns
<KnowledgeSnippet title="Production Strategy">
### Virtual Nodes (VNodes)
To prevent "Hot Spots" where one powerful server gets the same amount of traffic as a weak one, we use **Virtual Nodes**. Each physical server is mapped to multiple points on the hash ring. This ensures a more uniform distribution of data and allows us to assign more VNodes to more powerful servers.
</KnowledgeSnippet>
