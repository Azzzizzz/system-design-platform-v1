## Introduction

Rate limiting is a strategy for limiting network traffic. It puts a cap on how often someone can repeat an action within a certain timeframe – for instance, trying to log into an account. 

## Problem & Context

In distributed systems, rate limiting is the principal defense against **resource exhaustion**. Without it, your system is vulnerable to:

- **Resource Starvation**: A few heavy users monopolizing all available capacity.
- **Cascading Failures**: Traffic spikes overwhelming one service and triggering a chain reaction of failures.
- **Cost Runaway**: Unexpected usage spikes leading to massive cloud infrastructure bills.
- **Brute Force & DDoS**: Attackers overwhelming endpoints with high-velocity requests.

## Core Theory: Algorithms

Several standard algorithms are used to implement rate limiting, each with different performance characteristics:

### 1. Token Bucket
Tokens are added to a "bucket" at a fixed rate. Each request consumes a token. This allows for **short bursts** while maintaining a steady average.

### 2. Leaky Bucket
Requests enter a buffer and leak out at a constant rate, smoothing out bursts into a **uniform flow**.

### 3. Fixed Window
Divides time into fixed slots. It's simple but can allow $2\times$ the limit at window boundaries.

### 4. Sliding Window
A more accurate approach that prevents the boundary problem by using a moving time window, often implemented using sorted sets in Redis.

In a production environment, rate limiting is typically implemented as a centralized middleware or sidecar to maintain global state.

## Real-World Application

### Level 1: Foundational (Single API Nodes)
Small apps often use simple middleware like `express-rate-limit`. In this tier, state is usually stored in local memory. This is effective for protecting a single node against basic script-kiddies but doesn't scale across multiple instances.

### Level 2: Enterprise (Global Scale)
Global platforms like **Stripe** or **GitHub** use a tiered Redis-based approach. They use Lua scripts to ensure atomicity when incrementing counters and apply different limits based on identity (e.g., Authenticated vs. Anonymous).

## Implementation Patterns

<KnowledgeSnippet title="Distributed Atomicity">
### Solving the Race Condition
In a distributed system, two gateways might check a counter simultaneously. To solve this, use **Redis Lua scripts**. This ensures the "check-and-increment" logic runs as a single atomic operation, preventing users from bypassing limits during high-concurrency spikes.
</KnowledgeSnippet>

## Interview Notes
<InterviewAnswer>
### How would you implement rate limiting in a distributed system?

1. **Deployment Strategy** Use a centralized sidecar or middleware at the `API Gateway` level (e.g., `Envoy`, `Kong`) to intercept requests before they hit internal services.
2. **Distributed Storage** Use `Redis` for storing global counters. It is fast, atomic, and handles `TTL` (Time-to-Live) automatically, making it perfect for fixed or sliding window resets across multiple nodes.
3. **Algorithm Protocol** The `Token Bucket` is preferred for most public APIs. It provides a simple way to allow for **burst traffic** while maintaining a long-term average rate.
4. **Distributed Challenges** Systems must account for eventual consistency and clock variance:
   - **Race Conditions** Use `Redis Lua scripts` or `INCR` to ensure atomicity when multiple gateway instances check and decrement the same counter.
   - **Clock Drift** Use a centralized store's time (like Redis `TIME`) rather than local server time to ensure consistent window boundaries.
5. **Observability & UX** Always return an `HTTP 429` (Too Many Requests) status code. Include a `Retry-After` header to communicate back-off strategies to clients gracefully.
</InterviewAnswer>
