## Introduction
A load balancer sits in front of your servers and distributes incoming client requests across multiple backend servers. It ensures no single server bears too much load.

## Why It Matters
Without load balancing, all traffic hits one server, creating a single point of failure and performance bottleneck. Load balancers are foundational to building scalable, highly available systems.

## Problem It Solves
- Uneven distribution of traffic across servers
- Downtime caused by single server failures
- Inability to horizontally scale a service

## Core Concept
Load balancers operate at Layer 4 (transport) or Layer 7 (application). L4 routes based on IP/port, while L7 inspects HTTP headers/URLs for smarter routing.

**Key Algorithms:**
- **Round Robin:** Requests cycle through servers sequentially
- **Least Connections:** Routes to server with fewest active connections
- **IP Hash:** Same client IP always routed to same server
- **Weighted:** Distribute proportionally based on server capacity

**Health Checks:** Periodic pings to verify server availability. Unhealthy servers are removed from the pool automatically.

**Sticky Sessions:** Some applications need session affinity — the LB ensures the same user always reaches the same server.

## Real-World Application

### Level 1: Foundational (Small/Medium Apps)
Small to medium applications often use managed load balancers like **AWS Application Load Balancer (ALB)** or a single **Nginx** instance. In this tier, the focus is on basic high availability and simple TLS termination. It’s perfect for stabilizing traffic for a handful of web nodes.

### Level 2: Enterprise (Global Scale)
Global platforms like **Netflix** or **Cloudflare** use a tiered approach:
- **Global Server Load Balancing (GSLB)**: Directs users to the nearest regional data center using Anycast or DNS.
- **Hardware/L4 Balancers**: High-throughput devices (like F5) or software (Maglev) handle millions of packets at the edge.
- **Service Mesh (L7)**: Tools like **Envoy** or **Istio** manage fine-grained routing between internal microservices.

## Implementation Patterns
<KnowledgeSnippet title="Production Strategy">
### High Availability for the Balancer
To avoid the "Single Point of Failure" trap, never deploy just one load balancer. Use **Active-Passive** pairs with a **Floating IP** (VIP). If the active node dies, the passive node takes over the IP instantly using a protocol like VRRP or Keepalived.
</KnowledgeSnippet>

## Common Questions (Q&A)
<FAQAccordion items={[
  {
    question: "What is the difference between L4 and L7 load balancing?",
    answer: "L4 operates at the transport layer (TCP/UDP) and routes based on IP and Port. It is extremely fast but blind to the content. L7 operates at the application layer (HTTP) and can route based on cookies, headers, or URL paths, allowing for smarter decisions like A/B testing or path-based routing."
  },
  {
    question: "How do health checks actually work?",
    answer: "The load balancer periodically sends 'probes' (like an HTTP GET /health) to each server. If a server fails to respond or returns an error code multiple times, the LB stops sending it traffic. Once the server starts responding correctly again, it is automatically put back into the rotation."
  },
  {
    question: "When should I use Sticky Sessions?",
    answer: "Only use them when your application is stateful (e.g., storing data in local memory). However, it is a 'code smell' in modern design. It's better to make your application stateless by moving session data to a shared cache like Redis, allowing any server to handle any request."
  }
]} />
