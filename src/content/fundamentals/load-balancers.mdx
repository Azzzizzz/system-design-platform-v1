## Introduction
A load balancer sits in front of your servers and distributes incoming client requests across multiple backend servers. It ensures no single server bears too much load.

## Problem & Context
Without load balancing, all traffic hits a single server, creating a single point of failure and a performance bottleneck. As traffic grows, you need a way to distribute that load horizontally across a fleet of servers while maintaining high availability.

## Core Theory: Algorithms & OSI Layer
Load balancers operate at either Layer 4 (Transport) or Layer 7 (Application). L4 routes based on IP and Port, while L7 inspects HTTP headers for smarter, content-aware routing.

### Key Algorithms:
- **Round Robin:** Requests cycle through servers sequentially.
- **Least Connections:** Routes to the server with the fewest active connections.
- **IP Hash:** Uses the client's IP to ensure they always reach the same server.
- **Weighted:** Distributes traffic proportionally based on server capacity.

## Architecture & Components
To avoid becoming a single point of failure, load balancers are typically deployed in pairs using a Floating IP (VIP).

- **Sticky Sessions:** Ensuring a user stays with the same server for stateful apps.
- **TLS Termination:** Handling encryption at the balancer to offload server CPU.

## Real-World Application

### Level 1: Foundational (Small/Medium Apps)
Small to medium applications often use managed load balancers like **AWS Application Load Balancer (ALB)** or a single **Nginx** instance. In this tier, the focus is on basic high availability and simple TLS termination.

### Level 2: Enterprise (Global Scale)
Global platforms like **Netflix** or **Cloudflare** use a tiered approach:
- **Global Server Load Balancing (GSLB)**: Directs users to the nearest regional data center using Anycast DNS.
- **Hardware/L4 Balancers**: High-throughput devices (like F5) handle millions of packets at the edge.
- **Service Mesh (L7)**: Tools like **Envoy** or **Istio** manage fine-grained routing between internal microservices.

## Implementation Patterns

<KnowledgeSnippet title="Production Strategy">
### High Availability for the Balancer
Never deploy just one load balancer. Use **Active-Passive** pairs with a **Floating IP**. If the active node fails, the passive node takes over the IP instantly using protocols like VRRP or Keepalived, ensuring zero downtime.
</KnowledgeSnippet>

## Interview Notes

<InterviewAnswer>
### How would you design a load balancer for 10M concurrent connections?

1. **Tiered Architecture** Use **L4 (Transport Layer)** load balancing (like `Maglev` or `AWS NLB`) for massive raw throughput, followed by an **L7 (Application Layer)** tier for complex routing (SSL termination, header-based logic).
2. **High Availability (HA)** Deploy LBs in **Active-Passive** or **Active-Active** pairs using `VRRP`. If the primary fails, the backup takes over the **Virtual IP (VIP)** via gratuitous ARP.
3. **State Management** Use `Consistent Hashing` or `IP Hash` for stateful apps. This ensures a user stays on the same server while minimizing reshuffling if a backend node goes down.
4. **Health Verification** Implement **Active Health Checks** that hit a specific `/health` endpoint. This must verify not just the process but database connectivity and downstream dependencies.
5. **Global Traffic Control** Use **Global Server Load Balancing (GSLB)** via DNS to route users to the nearest healthy region, providing low latency and regional disaster recovery.
</InterviewAnswer>
