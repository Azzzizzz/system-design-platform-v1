## Introduction
In a distributed system, data is often replicated across multiple nodes for availability. **Consistency Models** define the rules that govern how and when a change made on one node becomes visible to other nodes.

## Problem & Context
Deciding on a consistency model is one of the most important architectural decisions. It is a direct trade-off between **Latency** and **Data Correctness**. Without a clear model, various nodes may return different versions of the same data, leading to "stale reads" or lost updates.

## Core Theory: Common Models

### Strong Consistency
The system guarantees that once a write is acknowledged, all subsequent reads will return that value. It usually requires a majority of nodes to confirm the write before responding.

### Eventual Consistency
The system guarantees that if no new updates are made, eventually all nodes will return the last updated value. This allows for massive scale and high availability at the cost of temporary staleness.

### Causal Consistency
A middle ground where the system ensures that operations that are "causally related" (e.g., a reply to a comment) are seen in the correct order by everyone.

## Real-World Application

### Level 1: Foundational (Social Media)
In a social media app like **Instagram**, "Likes" and "Comments" are often **Eventually Consistent**. If you like a photo, your friend might not see the updated count for a few seconds. This delay is acceptable because the cost of "being wrong" for a moment is zero.

### Level 2: Enterprise (Financial Systems)
For an e-commerce giant like **Amazon**, the **Checkout process** requires **Strong Consistency**. The system must ensure that an item is actually in stock and lock it before confirming the sale. Using eventual consistency here could lead to overselling inventory.

## Implementation Patterns

<KnowledgeSnippet title="Decision Framework">
### When to use Strong vs Eventual
Use **Strong Consistency** when data integrity is critical and the number of writers is small. Use **Eventual Consistency** when read-availability and low latency are prioritized over immediate correctness across all nodes.
</KnowledgeSnippet>

## Interview Notes

<InterviewAnswer>
### How do you handle "Read-Your-Writes" consistency in a distributed follower setup?

1. **Replication Lag** In **Leader-Follower** setups, async replication causes a delay. If a user writes to the leader and reads from a lagging follower, their own data appears to be missing.
2. **Leader-Only Routing** Route all reads for a user's *own* data (e.g., their user profile) to the **Leader Node**, while routing generic content (e.g., global feeds) to followers.
3. **Sticky Sessions** Pin a user's session to a specific follower at the **Load Balancer** level. This ensures they consistently see progress from at least one node's perspective.
4. **High-Water Marks** The client tracks its last update's `Version Tag`. The follower must confirm it has reached this version before serving the request, or it must redirect the read to the leader.
5. **Caching Layer** Use an atomic cache like `Redis` for the most recent writes. This provides a fast, consistent "short-term memory" until the permanent database replicas catch up.
</InterviewAnswer>
